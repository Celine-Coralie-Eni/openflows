# Architecture Refactor: Gateway + ReAct Nexus + Command Interpreter

## Problem Statement

The current communication architecture has several structural issues:

1. **Monolithic config**: `ChatConfig` bakes all channel-specific fields (slack, discord, whatsapp tokens/IDs) into a single struct — adding a new channel means modifying `ChatConfig`, `HumanChannel::new()`, and `ChannelType` enum
2. **No true gateway abstraction**: `HumanChannel` is a concrete aggregator, not a proper pluggable gateway — channels are hardcoded at construction time
3. **Nexus mixes orchestration + communication**: Command interpretation (`interpret_message`, `try_pattern_match`, `interpret_with_llm`) is embedded directly in `NexusNode`, coupling the orchestrator role with the communication parsing role
4. **Single-shot LLM decision, not ReAct**: Nexus calls `AgentRunner::run()` once per flow step — there's no iterative Reason→Act→Observe loop that would let it process a human chat message, reason about it, act on the system, observe the result, and continue
5. **No RAG integration point**: No way to inject knowledge/context for interpreting natural language commands

## Proposed Architecture

### 1. Gateway Crate (`nexus-gateway`) — replaces `nexus-chat`

Replace the current `nexus-chat` with a proper gateway abstraction that supports pluggable channel plugins.

#### `ChannelPlugin` trait

```rust
#[async_trait]
pub trait ChannelPlugin: Send + Sync {
    /// Unique identifier for this channel type (e.g. "slack", "discord")
    fn channel_id(&self) -> &str;

    /// Start listening for inbound messages. Pushes to `tx`.
    /// Returns a shutdown handle.
    async fn start_listener(
        &self,
        tx: mpsc::Sender<InboundMessage>,
        shutdown: watch::Receiver<bool>,
    ) -> Result<()>;

    /// Send an outbound message through this channel
    async fn send(&self, msg: &OutboundMessage) -> Result<()>;

    /// Ask a human a question and wait for response (optional)
    async fn ask_human(
        &self,
        question: &str,
        options: &[&str],
        ticket_id: &str,
        timeout_secs: u64,
    ) -> Option<String>;
}
```

#### `InboundMessage` / `OutboundMessage` (normalized)

Replace `HumanMessage`/`NexusMessage` with channel-agnostic message types:

```rust
pub struct InboundMessage {
    pub message_id: String,
    pub channel_id: String,       // which plugin sent it ("slack", "discord", ...)
    pub user_id: String,
    pub conversation_id: String,  // channel/thread/group ID
    pub text: String,
    pub timestamp: DateTime<Utc>,
    pub metadata: serde_json::Value,  // channel-specific extras
}

pub struct OutboundMessage {
    pub message_type: OutboundMessageType,
    pub target_channel: Option<String>,  // None = all channels
    pub target_conversation: Option<String>,
    pub content: String,
    pub ticket_id: Option<String>,
    pub worker_id: Option<String>,
    pub metadata: serde_json::Value,
}
```

#### `Gateway` struct

```rust
pub struct Gateway {
    plugins: HashMap<String, Box<dyn ChannelPlugin>>,
    inbound_tx: mpsc::Sender<InboundMessage>,
    inbound_rx: mpsc::Receiver<InboundMessage>,
    store: SharedStore,
}
```

- `Gateway::new()` creates an empty gateway
- `Gateway::register_plugin(plugin)` adds a channel plugin by its `channel_id()`
- `Gateway::start_listeners()` spawns listener tasks for all plugins
- `Gateway::recv_inbound()` returns the next inbound message (or polls)
- `Gateway::send(msg)` routes outbound messages to the correct plugin(s)
- `Gateway::broadcast(msg)` sends to all plugins
- `Gateway::shutdown()` signals all listeners to stop

#### `GatewayConfig` — dynamic, per-channel

Replace monolithic `ChatConfig` with a per-plugin config approach:

```rust
pub struct GatewayConfig {
    pub enabled: bool,
    pub dev_mode: bool,
    pub channels: HashMap<String, serde_json::Value>,  // channel_id → config blob
}
```

Each plugin reads its own config from the `channels` map. Environment variables are mapped by convention: `NEXUS_GATEWAY_SLACK_*`, `NEXUS_GATEWAY_DISCORD_*`, etc.

#### Channel plugin implementations

Each channel becomes a struct implementing `ChannelPlugin`:

- `SlackPlugin` — wraps current `SlackClient` + poller logic
- `DiscordPlugin` — wraps current `DiscordClient` + `discord_gateway` logic
- `WhatsAppPlugin` — wraps current `WhatsAppClient`
- `MockPlugin` — wraps current `MockChatClient` (dev mode)

These live as modules within `nexus-gateway` for now (same crate). The trait makes it trivial to extract them into separate crates later.

### 2. Command Interpreter (new module in `nexus-gateway`)

pub struct CommandInterpreter {

### 3. ReAct Nexus (refactor `agent-nexus`)

Refactor `NexusNode` to follow the ReAct (Reasoning + Acting) pattern when processing human messages, while retaining its existing `Node` trait integration with the `Flow`.

#### ReAct Loop

Currently, Nexus does:
1. `prep()` — gather state, process human commands (side-effect in prep!)
2. `exec()` — single LLM call
3. `post()` — execute decision

The ReAct refactor introduces a `ReActLoop` that Nexus uses when a human message arrives:

```
Observe:  Receive InboundMessage from Gateway
Reason:   LLM reasons about the message + system state
Act:      Execute an action (pause ticket, answer question, etc.)
Observe:  Check result of action (read store, check ticket status)
Reason:   LLM reasons about whether to continue or respond
Act:      Send response back to human, or take further action
...
```

#### `ReActStep` and `ReActLoop`

```rust
pub enum ReActStep {
    Observe { input: String },
    Reason { thought: String },
    Act { action: SystemCommand },
    Respond { message: OutboundMessage },
    Done,
}

pub struct ReActLoop {
    runner: AgentRunner,
    persona: AgentPersona,
    gateway: Gateway,
    knowledge_store: Box<dyn KnowledgeStore>,
    max_iterations: usize,
}
```

The ReAct loop:
1. **Observe**: Gather current system state (tickets, workers, PRs) + the inbound message
2. **Reason**: Call LLM with the observation + RAG context to decide what to do
3. **Act**: Execute the decided action against the system (modify store, call GitHub, etc.)
4. **Observe**: Check the result (did the ticket get paused? is the worker now idle?)
5. **Reason**: Decide if more action is needed or if a response should be sent
6. **Respond/Loop**: Send response back through Gateway, or continue the loop

This replaces the current `process_human_commands` + `interpret_message` flow in `NexusNode` with a more intelligent, iterative process.

#### Integration with existing Flow

`NexusNode` retains its `Node` implementation. The ReAct loop is triggered inside `prep()` when human messages are pending:

```rust
async fn prep(&self, store: &SharedStore) -> Result<Value> {
    // ... existing sync logic ...

    // Process human messages via ReAct loop
    while let Some(msg) = self.gateway.try_recv_inbound().await {
        let react_loop = ReActLoop::new(...);
        react_loop.run(msg, store).await;
    }

    // ... existing state gathering ...
}
```

The orchestration decision (assign work, merge PRs, etc.) still goes through `exec()` → `post()` as before. The ReAct loop is specifically for handling human communication.

### 5. Migration Strategy

The refactor should be done incrementally to avoid breaking the build:

**Phase 1: Create `nexus-gateway` crate (parallel with `nexus-chat`)**
- Create the new crate with `ChannelPlugin` trait, `Gateway`, `InboundMessage`, `OutboundMessage`, `SystemCommand`, `CommandInterpreter`, `KnowledgeStore` stub
- Port channel implementations: `SlackPlugin`, `DiscordPlugin`, `WhatsAppPlugin`, `MockPlugin`
- Write tests for Gateway and each plugin
- At this point both `nexus-chat` and `nexus-gateway` exist

**Phase 2: Implement ReAct loop in `agent-nexus`**
- Add `react_loop.rs` and `command_executor.rs` modules
- Implement `ReActLoop` using `AgentRunner` + `Gateway` + `KnowledgeStore`
- Move `interpret_message` / `try_pattern_match` / `interpret_with_llm` logic into `CommandInterpreter`
- Move `pause_ticket`, `resume_ticket`, `block_worker`, `reroute_work` into `CommandExecutor`
- Keep `NexusNode` working with both old `HumanChannel` and new `Gateway` (feature flag or wrapper)

**Phase 3: Switch `agent-nexus` to use `nexus-gateway`**
- Change `NexusNode::with_chat()` to `NexusNode::with_gateway()`
- Update `binary/src/main.rs` to use `Gateway` instead of `ChatConfig` + `run_chat_loop`
- Remove `nexus-chat` from workspace
- Update all tests

### 6. Key Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Plugin registration | Dynamic HashMap in Gateway | Allows runtime channel addition, config-driven |
| Config format | `HashMap<String, Value>` per channel | Each plugin owns its config schema; no monolithic struct |
| ReAct loop location | `nexus-gateway` crate | ReAct is a communication pattern, not just Nexus-specific |
| RAG implementation | Trait + Stub | Future-proof; stub allows testing without vector DB |
| Backward compat | 3-phase migration | Never break the build; both systems coexist during transition |
| Channel listener model | Each plugin owns its listener task | Consistent with current Discord Gateway pattern |

### 7. Resolved Decisions

1. **Channel plugins in same crate for now**: Keeping them in `nexus-gateway/src/channels/`. The `ChannelPlugin` trait boundary makes extraction to separate crates trivial later.

2. **ReAct loop uses same `AgentRunner`**: It reuses the same LLM client and MCP session. The difference is the prompt: orchestration uses `nexus.agent.md`, ReAct uses a command-interpreter prompt.

3. **Gateway + SharedStore interaction**: The Gateway stores pending inbound messages in SharedStore (same `KEY_HUMAN_MESSAGES` key). This maintains compatibility with the existing flow architecture. The ReAct loop reads from the Gateway's inbound channel directly.

4. **ReAct loop lives in `nexus-gateway` crate**: ReAct is a communication/reasoning pattern, not Nexus-specific. Other agents could use it later. The `react.rs` module is part of the gateway crate.

5. **3-phase migration**: Both `nexus-chat` and `nexus-gateway` coexist during transition. Never breaks the build.
