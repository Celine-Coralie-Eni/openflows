# Plan: Codex CLI Backend Integration for AgentFlow

## Context

AgentFlow already has a `CliBackend` enum (`Claude` / `Codex`) and partial Codex support in `ProcessManager`. However, the Codex integration is skeletal — it only handles `codex exec --full-auto` with basic env var injection. The Codex CLI has a rich **harness architecture** (App Server JSON-RPC, hooks, skills, subagents, plugins, AGENTS.md, permission profiles) that we should leverage to match the depth of our Claude Code integration.

This plan maps every Codex harness primitive to its AgentFlow equivalent and specifies what code changes are needed to make Codex a first-class backend.

## Key Codex Harness Primitives (from docs)

| Codex Primitive | Description | AgentFlow Equivalent |
|---|---|---|
| **App Server** | JSON-RPC over stdio (bidirectional). Thread/Turn/Item lifecycle. | ProcessManager + watcher (our event loop) |
| **codex exec** | Non-interactive one-shot: `codex exec --full-auto "<prompt>"` | `spawn_forge_with_backend(Codex)` |
| **Hooks** | Lifecycle scripts: SessionStart, PreToolUse, PostToolUse, PermissionRequest, Stop, etc. | `orchestration/plugin/hooks/` (shell scripts) |
| **Skills** | SKILL.md with progressive disclosure, scripts/, references/ | `orchestration/plugin/skills/` (already SKILL.md format!) |
| **Subagents** | Custom .toml agent definitions under `.codex/agents/` | `orchestration/agent/agents/*.agent.md` personas |
| **Plugins** | Bundle skills + hooks + MCP + apps in installable package | `orchestration/plugin/.codex-plugin/plugin.json` (exists!) |
| **AGENTS.md** | Layered instruction discovery (global → project → per-directory) | `orchestration/agent/agents/*.agent.md` (already consumed by Claude) |
| **Permissions** | Permission profiles with filesystem + network rules | `.claude/settings.json` (Claude side) — needs `.codex/` equivalent |
| **Config** | `config.toml` at `~/.codex/` and `.codex/` with profiles | `.claude/settings.json` (Claude side) — needs `.codex/` equivalent |
| **MCP** | MCP servers in config.toml `[mcp_servers]` section | `.claude/mcp.json` (Claude side) — needs `.codex/` equivalent |

## Architecture Decision: Integration Modes

Codex offers three integration modes for programmatic control:

1. **`codex exec`** (current implementation) — fire-and-forget, non-interactive. Good for SENTINEL (ephemeral). Limited for FORGE (long-running, needs monitoring).

2. **App Server** (recommended) — bidirectional JSON-RPC over stdio. Gives us:
   - Turn/Item lifecycle events (progress streaming)
   - Approval handling through protocol (instead of `--full-auto` bypass)
   - Thread persistence (resume conversations)
   - Diff emission during work

3. **MCP Server** (`codex mcp-server`) — exposes Codex as an MCP tool. Limited to MCP semantics.

**Decision**: Use `codex exec` for SENTINEL (ephemeral, one-shot) and **App Server** for FORGE (long-running, needs monitoring/resume). This gives us the richest harness integration.

## Implementation Plan

### Phase 1: Provisioning — Generate `.codex/` Config Files

When `cli_backend == Codex`, the Provisioner must generate Codex-native config instead of (or alongside) `.claude/` config.

#### 1.1 Add `Provisioner::provision_codex_config()` method

**File**: `crates/pair-harness/src/provision.rs`

Generate for each worktree/shared dir when backend is Codex:

- `.codex/config.toml` — sandbox mode, approval policy, model, MCP servers, hooks reference
- `.codex/agents/forge.toml` / `.codex/agents/sentinel.toml` — custom agent definitions that **read** persona from `orchestration/agent/agents/*.agent.md` (single source of truth, no duplication)
- `.agents/skills/` — symlink our existing SKILL.md files (Codex scans `.agents/skills/`)
- AGENTS.md at worktree root — **read** persona from `orchestration/agent/agents/{role}.agent.md` and inject into AGENTS.md (Codex reads AGENTS.md first)

Example `.codex/config.toml` for FORGE worktree:
```toml
approval_policy = "on-request"
sandbox_mode = "workspace-write"

[sandbox_workspace_write]
network_access = true

[mcp_servers.github]
command = "npx"
args = ["-y", "@modelcontextprotocol/server-github"]

[mcp_servers.github.env]
GITHUB_PERSONAL_ACCESS_TOKEN = "${SPRINTLESS_GITHUB_TOKEN}"

[mcp_servers.filesystem]
command = "npx"
args = ["-y", "@modelcontextprotocol/server-filesystem", "${SPRINTLESS_WORKTREE}", "${SPRINTLESS_SHARED}"]
```

Example `.codex/agents/forge.toml` — **reads persona from existing agent.md, does NOT duplicate content**:
```toml
name = "forge"
description = "Senior engineer. Writes code, tests, opens PRs."
model = "gpt-5.4"
sandbox_mode = "workspace-write"
# developer_instructions is populated by reading orchestration/agent/agents/forge.agent.md at provision time
```

The provisioner reads `orchestration/agent/agents/{id}.agent.md` at provision time and injects its content into `developer_instructions`. This keeps `agent.md` files as the single source of truth — Codex config is generated, not maintained separately.

#### 1.2 Add `Provisioner::provision_pair_codex()` orchestration method

Called from `provision_pair()` when `cli_backend == Codex`. Skips `.claude/` generation and generates `.codex/` instead.

#### 1.3 Update `ForgeSentinelPair::provision_config()` 

**File**: `crates/pair-harness/src/pair.rs`

Pass `cli_backend` to provisioner so it generates the right config format.

### Phase 2: Process Spawning — Fix Codex Paths & Add App Server Mode

#### 2.1 Fix `spawn_forge_for_pr()` — currently hardcoded to Claude

**File**: `crates/pair-harness/src/process.rs` (line ~572)

The method creates a `Command::new(&self.claude_path)` directly. It needs to use `self.build_cli_command(backend, ...)` and `self.inject_cli_env()` like `spawn_forge_with_backend()` does.

#### 2.2 Add App Server spawn mode for FORGE

New method `spawn_forge_app_server()`:

```rust
pub async fn spawn_forge_app_server(
    &self,
    pair_id: &str,
    ticket_id: &str,
    worktree: &Path,
    shared: &Path,
) -> Result<AppServerClient>
```

- Spawns `codex app-server` as a child process
- Returns an `AppServerClient` that speaks JSON-RPC over stdio
- The client can: `initialize()`, `create_thread()`, `submit_turn(prompt)`, `recv_notifications()`, `send_approval(allow/deny)`
- On FORGE exit or stall, the client can resume the thread

#### 2.3 Define `AppServerClient` struct

**New file**: `crates/pair-harness/src/app_server.rs`

```rust
pub struct AppServerClient {
    child: Child,
    stdin: BufWriter<ChildStdin>,
    stdout: BufReader<ChildStdout>,
    next_id: u64,
}

impl AppServerClient {
    pub async fn initialize(&mut self) -> Result<InitializeResult>;
    pub async fn create_thread(&mut self) -> Result<String>; // thread_id
    pub async fn submit_turn(&mut self, thread_id: &str, prompt: &str) -> Result<()>;
    pub async fn recv_notification(&mut self) -> Result<Notification>; // Item/Turn/Thread events
    pub async fn send_approval(&mut self, request_id: u64, decision: ApprovalDecision) -> Result<()>;
    pub async fn close(&mut self) -> Result<()>;
}
```

The notification stream maps directly to our `FsEvent`-driven event loop but is richer:
- `item/started` + `item/completed` → FORGE started/finished a tool call
- `turn/completed` → FORGE finished its response
- Approval requests → we can auto-approve (since we use `workspace-write` sandbox)

#### 2.4 Integrate App Server client into the pair event loop

**File**: `crates/pair-harness/src/pair.rs`

When `cli_backend == Codex`, the event loop uses `AppServerClient` instead of raw `Child`:
- On `turn/completed`, check if FORGE wrote PLAN.md / STATUS.json (same filesystem checks)
- On approval requests, auto-approve if within sandbox policy
- On `SubagentStop`, feed continuation prompt (like our HANDOFF.md resume)

### Phase 3: Subagent Definitions — Generate Codex Agent TOML from Existing Personas

**Key principle**: `orchestration/agent/agents/*.agent.md` are the single source of truth for all personas. Codex `.toml` files are GENERATED from these — never manually maintained or duplicated.

#### 3.1 Generate `.codex/agents/*.toml` by reading existing agent.md files

For each active agent in registry.json, the provisioner reads `orchestration/agent/agents/{id}.agent.md` and generates a corresponding `.codex/agents/<id>.toml`:

```rust
fn generate_codex_agent_toml(
    agent_entry: &RegistryEntry,
    agent_md_path: &Path,
) -> Result<String> {
    let persona = std::fs::read_to_string(agent_md_path)?;
    // Extract frontmatter fields (id, role, cli, etc.) from agent.md
    // Map to Codex TOML fields
    Ok(format!(
        r#"name = "{id}"
description = "{role} agent for AgentFlow orchestration"
model = "{model}"
sandbox_mode = "{sandbox}"
developer_instructions = \"\"\"{persona}\"\"\"
"#,
        id = agent_entry.id,
        role = extract_role_from_frontmatter(&persona),
        model = agent_entry.model_backend.as_deref().unwrap_or("gpt-5.4"),
        sandbox = if agent_entry.id == "sentinel" { "read-only" } else { "workspace-write" },
        persona = persona,
    ))
}
```

The frontmatter from `forge.agent.md` (e.g., `role: builder`, `cli: claude`) is parsed and mapped to Codex fields. The full persona body becomes `developer_instructions`.

#### 3.2 Set `[agents]` global config


### Phase 4: Hooks — Wire Existing Shell Hooks to Codex Hook Schema

#### 4.1 Generate `.codex/hooks.json` from our existing hooks

**File**: `orchestration/plugin/hooks/hooks.json` already defines hooks per agent role. We need to map these to Codex's hook schema:

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup|resume",
        "hooks": [
          {
            "type": "command",
            "command": "/abs/path/to/orchestration/plugin/hooks/forge/session_start.sh",
            "statusMessage": "Loading FORGE session context"
          }
        ]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "/abs/path/to/orchestration/plugin/hooks/forge/pre_bash_guard.sh",
            "statusMessage": "Checking Bash command"
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "/abs/path/to/orchestration/plugin/hooks/forge/stop_require_artifact.sh",
            "timeout": 30
          }
        ]
      }
    ]
  }
}
```

Key Codex hook events to map:
| Codex Event | Our Hook | Purpose |
|---|---|---|
| `SessionStart` | `session_start.sh` | Inject context, set env |
| `PreToolUse(Bash)` | `pre_bash_guard.sh` | Block dangerous commands |
| `PreToolUse(apply_patch)` | `pre_write_check.sh` | Validate writes |
| `PostToolUse(Bash)` | `post_write_lint.sh` | Lint after changes |
| `PreCompact` | `pre_compact_handoff.sh` | Write HANDOFF.md before context reset |
| `Stop` | `stop_require_artifact.sh` | Ensure STATUS.json written before exit |
| `SubagentStart` | N/A | Inject subagent context |
| `SubagentStop` | N/A | Validate subagent output |

#### 4.2 Generate hooks.json in provisioning

The provisioner should resolve absolute paths to the hook scripts (they must be absolute for Codex) and write the Codex-format hooks.json.

### Phase 6: AGENTS.md — Layered Instruction Discovery

Codex reads AGENTS.md with a layered discovery system (global → project → per-directory). Our `forge.agent.md` etc. are Claude-specific files. We need to:

#### 6.1 Write AGENTS.md to worktree root — **read** from existing agent.md

When `cli_backend == Codex`, provisioner **reads** the appropriate `orchestration/agent/agents/{role}.agent.md` and writes its content as AGENTS.md in the worktree root. This is a format conversion, not content creation:

- FORGE worktree gets AGENTS.md built from `forge.agent.md` + standards (CODING.md, SECURITY.md)
- SENTINEL shared dir gets AGENTS.md built from `sentinel.agent.md` + REVIEW.md

The agent.md files remain the source of truth. AGENTS.md is generated at provision time by reading them.

#### 6.2 Write global AGENTS.md to `~/.codex/AGENTS.md`

For cross-repository defaults (working agreements, tool preferences).

### Phase 8: Environment & Model Routing

#### 8.1 Fix Codex model routing

Currently `inject_cli_env(Codex)` passes `OPENAI_BASE_URL` and `OPENAI_API_KEY`. But when using a proxy like Fireworks/LiteLLM:

- For Claude backend: `ANTHROPIC_BASE_URL` → proxy, `ANTHROPIC_API_KEY` → routing_key
- For Codex backend: `OPENAI_BASE_URL` → proxy, `OPENAI_API_KEY` → routing_key

The `model_backend` from registry (e.g., `accounts/fireworks/models/kimi-k2p5`) needs to be passed as the `-m` flag to `codex exec`.

#### 8.2 Add `CODEX_HOME` support

Set `CODEX_HOME` env var per-worktree so each pair has isolated Codex config:
```
CODEX_HOME=/path/to/worktree/.codex-home
```

This prevents config collisions between parallel forge-1/forge-2 workers.

### Phase 9: Security — Permission Profiles for Codex

#### 9.1 Generate Codex permission profiles

Replace `.claude/settings.json` (Claude's permission model) with Codex's permission profile system:

```toml
default_permissions = "workspace-write"

[permissions.workspace-write.filesystem]
":minimal" = "read"

[permissions.workspace-write.filesystem.":workspace_roots"]
"." = "write"
"**/*.env" = "deny"

[permissions.workspace-write.network]
enabled = true

[permissions.workspace-write.network.domains]
"api.github.com" = "allow"
"*.github.com" = "allow"
```

For SENTINEL (read-only):
```toml
default_permissions = "read-only"

[permissions.read-only.filesystem]
":minimal" = "read"

[permissions.read-only.filesystem.":workspace_roots"]
"." = "read"
```

## Implementation Order

1. **Phase 2.1** (fix `spawn_forge_for_pr`) — Quick win, existing bug
2. **Phase 1** (provisioning) — Foundation for everything else
3. **Phase 8** (model routing) — Makes Codex actually work with our proxy
4. **Phase 7** (output parsing) — Needed to consume Codex results
5. **Phase 6** (AGENTS.md) — Gives Codex agents their instructions
6. **Phase 4** (hooks) — Safety guardrails
7. **Phase 5** (skills) — Reuse our existing skills
8. **Phase 3** (subagents) — Enables Codex's parallel agent spawning
9. **Phase 9** (permissions) — Security hardening
10. **Phase 2.2-2.4** (App Server) — Advanced: bidirectional monitoring
11. **Phase 10** (docs) — Last, after implementation stabilizes

## Open Questions

1. Should the App Server integration be Phase 2 (early) or deferred? It's the richest integration but also the most complex. The `codex exec` mode works today for the basic case.
2. Should we generate both `.claude/` and `.codex/` config always (so agents can be mixed per-pair), or only generate the config for the selected backend?
3. The `model_backend` field in registry currently maps to LiteLLM routing keys. For Codex, the model is specified via `-m` flag or `OPENAI_MODEL` env var. Should we add a `codex_model` field to registry, or reuse `model_backend` with a naming convention?
