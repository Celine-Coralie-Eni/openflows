# Plan: Codex CLI Backend Integration for AgentFlow

## Context

AgentFlow already has a `CliBackend` enum (`Claude` / `Codex`) and partial Codex support in `ProcessManager`. However, the Codex integration is skeletal — it only handles `codex exec --full-auto` with basic env var injection. The Codex CLI has a rich **harness architecture** (App Server JSON-RPC, hooks, skills, subagents, plugins, AGENTS.md, permission profiles) that we should leverage to match the depth of our Claude Code integration.

This plan maps every Codex harness primitive to its AgentFlow equivalent and specifies what code changes are needed to make Codex a first-class backend.

## Key Codex Harness Primitives (from docs)

| Codex Primitive | Description | AgentFlow Equivalent |
|---|---|---|
| **App Server** | JSON-RPC over stdio (bidirectional). Thread/Turn/Item lifecycle. | ProcessManager + watcher (our event loop) |
| **codex exec** | Non-interactive one-shot: `codex exec --full-auto "<prompt>"` | `spawn_forge_with_backend(Codex)` |
| **Hooks** | Lifecycle scripts: SessionStart, PreToolUse, PostToolUse, PermissionRequest, Stop, etc. | `orchestration/plugin/hooks/` (shell scripts) |
| **Skills** | SKILL.md with progressive disclosure, scripts/, references/ | `orchestration/plugin/skills/` (already SKILL.md format!) |
| **Subagents** | Custom .toml agent definitions under `.codex/agents/` | `orchestration/agent/agents/*.agent.md` personas |
| **Plugins** | Bundle skills + hooks + MCP + apps in installable package | `orchestration/plugin/.codex-plugin/plugin.json` (exists!) |
| **AGENTS.md** | Layered instruction discovery (global → project → per-directory) | `orchestration/agent/agents/*.agent.md` (already consumed by Claude) |
| **Permissions** | Permission profiles with filesystem + network rules | `.claude/settings.json` (Claude side) — needs `.codex/` equivalent |
| **Config** | `config.toml` at `~/.codex/` and `.codex/` with profiles | `.claude/settings.json` (Claude side) — needs `.codex/` equivalent |
| **MCP** | MCP servers in config.toml `[mcp_servers]` section | `.claude/mcp.json` (Claude side) — needs `.codex/` equivalent |

## Architecture Decision: Integration Mode

Codex offers three integration modes for programmatic control:

1. **`codex exec`** — fire-and-forget, non-interactive one-shot execution.
2. **App Server** — bidirectional JSON-RPC over stdio (complex, deferred).
3. **MCP Server** (`codex mcp-server`) — exposes Codex as an MCP tool.

**Decision**: Use `codex exec --full-auto` with **hooks** for both FORGE and SENTINEL. Our existing hook-based lifecycle management (SessionStart, PreToolUse, PostToolUse, Stop, SubagentStart, SubagentStop) combined with filesystem watching (STATUS.json, PLAN.md, HANDOFF.md) covers all orchestration decision points. The App Server adds significant complexity for marginal benefit — deferred until we need real-time token streaming or finer-grained control.

### Phase 3: Subagent Definitions — Generate Codex Agent TOML from Existing Personas

**Key principle**: `orchestration/agent/agents/*.agent.md` are the single source of truth for all personas. Codex `.toml` files are GENERATED from these — never manually maintained or duplicated.

#### 3.1 Generate `.codex/agents/*.toml` by reading existing agent.md files

For each active agent in registry.json, the provisioner reads `orchestration/agent/agents/{id}.agent.md` and generates a corresponding `.codex/agents/<id>.toml`:

```rust
fn generate_codex_agent_toml(
    agent_entry: &RegistryEntry,
    agent_md_path: &Path,
) -> Result<String> {
    let persona = std::fs::read_to_string(agent_md_path)?;
    // Extract frontmatter fields (id, role, cli, etc.) from agent.md
    // Map to Codex TOML fields
    Ok(format!(
        r#"name = "{id}"
description = "{role} agent for AgentFlow orchestration"
model = "{model}"
sandbox_mode = "{sandbox}"
developer_instructions = """{persona}"""
"#,
        id = agent_entry.id,
        role = extract_role_from_frontmatter(&persona),
        model = agent_entry.model_backend.as_deref().unwrap_or("gpt-5.4"),
        sandbox = if agent_entry.id == "sentinel" { "read-only" } else { "workspace-write" },
        persona = persona,
    ))
}
```

The frontmatter from `forge.agent.md` (e.g., `role: builder`, `cli: claude`) is parsed and mapped to Codex fields. The full persona body becomes `developer_instructions`.


#### 4.2 Create SubagentStart hook scripts

**New files**: `orchestration/plugin/hooks/forge/subagent_start.sh`, `orchestration/plugin/hooks/sentinel/subagent_start.sh`

When Codex spawns a subagent, this hook runs to:
- Inject the subagent's identity (which agent type, what role)
- Set up subagent-specific environment variables
- Read the subagent's persona from `orchestration/agent/agents/{subagent_id}.agent.md`
- Log subagent spawn to shared event log

Example `orchestration/plugin/hooks/forge/subagent_start.sh`:
```bash
#!/bin/bash
# Runs when FORGE spawns a subagent
# This hook injects subagent context and sets up environment
#
# Environment:
#   SPRINTLESS_SUBAGENT_ID - the subagent identifier (e.g., "forge-test-writer")
#   SPRINTLESS_SUBAGENT_ROLE - the subagent's role (e.g., "test-writer")
#   SPRINTLESS_SUBAGENT_TASK - the specific task assigned to subagent
#   SPRINTLESS_SHARED - the shared directory
#   SPRINTLESS_WORKTREE - the worktree directory

SUBAGENT_ID="${SPRINTLESS_SUBAGENT_ID}"
SUBAGENT_ROLE="${SPRINTLESS_SUBAGENT_ROLE}"
SUBAGENT_TASK="${SPRINTLESS_SUBAGENT_TASK}"
SHARED="${SPRINTLESS_SHARED}"
WORKTREE="${SPRINTLESS_WORKTREE}"

# Log subagent spawn
echo "[$(date -u +%Y-%m-%dT%H:%M:%SZ)] forge subagent_start id=${SUBAGENT_ID} role=${SUBAGENT_ROLE}" \
  >> "${SHARED}/../events.log"

# Read subagent persona if it exists
PERSONA_FILE="/abs/path/to/orchestration/agent/agents/${SUBAGENT_ID}.agent.md"
if [ -f "$PERSONA_FILE" ]; then
  echo "Loading subagent persona from ${PERSONA_FILE}"
  # Inject persona content into subagent context
  export SUBAGENT_PERSONA=$(cat "$PERSONA_FILE")
else
  echo "WARNING: No persona file found for subagent ${SUBAGENT_ID}"
  echo "Using default role-based instructions for: ${SUBAGENT_ROLE}"
fi

# Set up subagent working directory
SUBAGENT_WORK_DIR="${WORKTREE}/.subagents/${SUBAGENT_ID}"
mkdir -p "$SUBAGENT_WORK_DIR"
export SPRINTLESS_SUBAGENT_WORK_DIR="$SUBAGENT_WORK_DIR"

echo "=========================================="
echo "  SUBAGENT STARTED: ${SUBAGENT_ID}"
echo "=========================================="
echo ""
echo "Role: ${SUBAGENT_ROLE}"
echo "Task: ${SUBAGENT_TASK}"
echo "Working Dir: ${SUBAGENT_WORK_DIR}"
echo ""
echo "IMPORTANT - Subagent Guidelines:"
echo "  1. Work ONLY in your assigned directory: ${SUBAGENT_WORK_DIR}"
echo "  2. Do NOT modify files outside your work directory"
echo "  3. Write results to ${SUBAGENT_WORK_DIR}/output.md"
echo "  4. Update ${SUBAGENT_WORK_DIR}/status.json when done"
echo "  5. Do NOT attempt to create PRs or push code"
echo ""

exit 0
```

#### 4.3 Create SubagentStop hook scripts

**New files**: `orchestration/plugin/hooks/forge/subagent_stop.sh`, `orchestration/plugin/hooks/sentinel/subagent_stop.sh`

When a subagent completes, this hook runs to:
- Validate the subagent produced required artifacts
- Collect subagent output into the shared directory
- Log subagent completion
- Block exit if artifacts are missing

Example `orchestration/plugin/hooks/forge/subagent_stop.sh`:
```bash
#!/bin/bash
# Runs when a FORGE subagent completes
# Validates subagent output and collects artifacts
#
# Environment:
#   SPRINTLESS_SUBAGENT_ID - the subagent identifier
#   SPRINTLESS_SUBAGENT_ROLE - the subagent's role
#   SPRINTLESS_SUBAGENT_WORK_DIR - the subagent's working directory
#   SPRINTLESS_SHARED - the shared directory

SUBAGENT_ID="${SPRINTLESS_SUBAGENT_ID}"
SUBAGENT_WORK_DIR="${SPRINTLESS_SUBAGENT_WORK_DIR}"
SHARED="${SPRINTLESS_SHARED}"

# Log subagent completion
echo "[$(date -u +%Y-%m-%dT%H:%M:%SZ)] forge subagent_stop id=${SUBAGENT_ID}" \
  >> "${SHARED}/../events.log"

# Check for required output artifacts
OUTPUT_FILE="${SUBAGENT_WORK_DIR}/output.md"
STATUS_FILE="${SUBAGENT_WORK_DIR}/status.json"

if [ ! -f "$OUTPUT_FILE" ]; then
  echo "ERROR: Subagent ${SUBAGENT_ID} did not produce output.md"
  echo "Subagent must write results to: ${OUTPUT_FILE}"
  exit 2
fi

if [ ! -f "$STATUS_FILE" ]; then
  echo "ERROR: Subagent ${SUBAGENT_ID} did not produce status.json"
  echo "Subagent must write status to: ${STATUS_FILE}"
  exit 2
fi

# Validate status.json has required fields
if command -v python3 &> /dev/null; then
  VALID=$(python3 -c "
import json, sys
try:
    s = json.load(open('${STATUS_FILE}'))
    required = ['status', 'subagent_id', 'task_completed']
    missing = [k for k in required if k not in s]
    if missing:
        print(f'missing: {missing}')
        sys.exit(1)
    if s['status'] not in ['COMPLETED', 'FAILED', 'PARTIAL']:
        print(f'invalid status: {s[\"status\"]}')
        sys.exit(1)
except Exception as e:
    print(str(e))
    sys.exit(1)
" 2>&1)
  if [ $? -ne 0 ]; then
    echo "status.json exists but is invalid: ${VALID}"
    exit 2
  fi
fi

# Copy subagent artifacts to shared directory
cp "$OUTPUT_FILE" "${SHARED}/subagent-${SUBAGENT_ID}-output.md"
cp "$STATUS_FILE" "${SHARED}/subagent-${SUBAGENT_ID}-status.json"

echo "=========================================="
echo "  SUBAGENT COMPLETED: ${SUBAGENT_ID}"
echo "=========================================="
echo ""
echo "Artifacts collected:"
echo "  - ${SHARED}/subagent-${SUBAGENT_ID}-output.md"
echo "  - ${SHARED}/subagent-${SUBAGENT_ID}-status.json"
echo ""

exit 0
```

Example `orchestration/plugin/hooks/sentinel/subagent_stop.sh`:
```bash
#!/bin/bash
# Runs when a SENTINEL subagent completes
# Validates subagent evaluation output
#
# Environment:
#   SPRINTLESS_SUBAGENT_ID - the subagent identifier
#   SPRINTLESS_SHARED - the shared directory

SUBAGENT_ID="${SPRINTLESS_SUBAGENT_ID}"
SHARED="${SPRINTLESS_SHARED}"

# Check for eval file
EVAL_FILE="${SHARED}/subagent-${SUBAGENT_ID}-eval.md"

if [ ! -f "$EVAL_FILE" ]; then
  echo "ERROR: Subagent ${SUBAGENT_ID} did not produce evaluation file"
  echo "Must write to: ${EVAL_FILE}"
  exit 2
fi

# Validate eval file has required sections
if ! grep -q "## Verdict" "$EVAL_FILE"; then
  echo "ERROR: ${EVAL_FILE} missing ## Verdict section"
  exit 2
fi

VERDICT=$(grep -A1 "## Verdict" "$EVAL_FILE" | tail -1 | tr -d ' ')
if [ "$VERDICT" != "APPROVED" ] && [ "$VERDICT" != "NEEDS_WORK" ]; then
  echo "ERROR: Invalid verdict in ${EVAL_FILE}: ${VERDICT}"
  echo "Must be APPROVED or NEEDS_WORK"
  exit 2
fi

echo "Subagent ${SUBAGENT_ID} evaluation complete: ${VERDICT}"
exit 0
```

The agent.md files remain the source of truth. AGENTS.md is generated at provision time by reading them.

### Phase 8: Environment & Model Routing

#### 8.1 Fix Codex model routing

Currently `inject_cli_env(Codex)` passes `OPENAI_BASE_URL` and `OPENAI_API_KEY`. But when using a proxy like Fireworks/LiteLLM:

- For Claude backend: `ANTHROPIC_BASE_URL` → proxy, `ANTHROPIC_API_KEY` → routing_key
- For Codex backend: `OPENAI_BASE_URL` → proxy, `OPENAI_API_KEY` → routing_key

The `model_backend` from registry (e.g., `accounts/fireworks/models/kimi-k2p5`) needs to be passed as the `-m` flag to `codex exec`.

#### 8.2 Add `CODEX_HOME` support

Set `CODEX_HOME` env var per-worktree so each pair has isolated Codex config:
```
CODEX_HOME=/path/to/worktree/.codex-home
```

This prevents config collisions between parallel forge-1/forge-2 workers.
